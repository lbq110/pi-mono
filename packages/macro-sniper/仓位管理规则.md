## 一、ATR 自适应仓位计算

### 核心公式

```
单标的上限 = 账户权益 × 20%
仓位金额  = min(单标的上限, 风险预算 ÷ 止损百分比)
```

其中：
- **单标的上限** = 账户权益 × `POSITION_MAX_PCT`（20%），随账户规模自动缩放
- **风险预算** = 账户权益 × 单笔风险比例（1%）
- **止损百分比** = K × ATR%（K = 2）

### 参数

| 参数 | 值 | 来源 |
|------|------|------|
| 单标的上限比例 | 20% | `POSITION_MAX_PCT = 0.20` |
| ATR 周期 | 14 日 | `ATR_PERIOD = 14` |
| ATR 倍数 K | 2 | `ATR_MULTIPLIER = 2` |
| 单笔风险 | 1% | `RISK_PER_TRADE = 0.01` |

### 单标的上限示例

| 账户权益 | 单标的上限 (20%) |
|------|------|
| $50,000 | $10,000 |
| $100,000 | $20,000 |
| $200,000 | $40,000 |

### ATR 计算方法

1. 从 `hourly_prices` 表按 UTC 日期聚合为日线 OHLC
2. 真实波幅 TR = max(H−L, |H−前收|, |L−前收|)
3. ATR = 最近 14 个 TR 的简单移动平均（SMA）
4. 需要至少 15 根日线（14 个 TR + 1 根基准），数据不足时跳过 ATR 调整，使用比例上限

### 仓位计算示例（账户 $100,000，上限 $20,000）

| 标的 | ATR% | 止损距离 (2×ATR%) | 风险预算 ($1,000) | ATR 计算仓位 | 实际仓位 |
|------|------|------|------|------|------|
| SPY | 1.2% | 2.4% | $1,000 | $41,667 | $20,000 (受上限约束) |
| BTC | 5.0% | 10.0% | $1,000 | $10,000 | $10,000 (ATR 约束) |
| UUP | 0.3% | 0.6% | $1,000 | $166,667 | $20,000 (受上限约束) |

> 高波动标的（BTC）由 ATR 自动缩小仓位；低波动标的由比例上限约束。账户增长时上限同步提高。

### 实现文件

- `src/analyzers/atr.ts` — `computeATR()`, `computeAllATRs()`, `atrPositionSize()`
- `src/executors/signal-scorer.ts` — `adjustNotionalForATR()`, Layer 2 调整

## 二、移动止损（吊灯止损 + 保本止损）

### 机制

将吊灯止损（Chandelier Exit）与保本止损（Breakeven Stop）合并为单一移动止损逻辑。

### 公式

**做多：**
```
止损价 = max(开仓成本, 最高价 − K × ATR)
```

**做空（UUP）：**
```
止损价 = min(开仓成本, 最低价 + K × ATR)
```

### 行为说明

| 阶段 | 止损位置 | 方法标记 |
|------|------|------|
| ATR 可用，价格未盈利 | 开仓成本（保本） | `breakeven` |
| ATR 可用，价格已盈利 | 最高价 − 2×ATR（吊灯） | `trailing_atr` |
| ATR 数据不足 | 开仓成本 × (1 − 8%) | `fallback_fixed` |

### 价格追踪

- `positions.high_water_mark` 列追踪仓位开立以来的最高价（做多）或最低价（做空）
- 每次仓位同步后自动更新（`updateHighWaterMarks()`）
- 止损触发后重置为 `null`

### 触发后处理

1. 立即市价平仓（Alpaca API `closePosition()`）
2. 写入 `risk_events` 表（`event_type = "stop_loss_l1"`）
3. 设置 24 小时冷却期（该标的在冷却期内禁止重新入场）
4. Slack 推送告警（标的、止损方式、亏损%、绝对亏损、冷却到期时间）

### 检查频率

- 每小时整点（:00）由 cron scheduler 自动触发
- CLI 手动触发：`macro-sniper risk check`

### 实现文件

- `src/executors/risk-manager.ts` — `computeTrailingStop()`, `checkStopLoss()`, `updateHighWaterMarks()`
- `src/db/schema.ts` — `positions.highWaterMark` 列

## 三、回撤分级管理

### 分级阈值

基于账户权益相对于历史最高点（Portfolio HWM）的回撤比例，自动调整全局风控乘数。

| 等级 | 条件 | 风控乘数 | 操作 |
|------|------|------|------|
| 🟢 **normal** | 回撤 < 5% | ×1.0 | 正常交易 |
| 🟡 **caution** | 回撤 ≥ 5% | ×0.5 | 所有仓位减半 |
| 🟠 **warning** | 回撤 ≥ 10% | ×0.25 | 所有仓位降至四分之一 |
| 🔴 **halt** | 回撤 ≥ 15% | ×0 | 停止一切交易，所有标的方向强制为 flat |

### 降级规则

- **自动降级**：当回撤比例超过阈值时立即降级（只降不升）
- 降级同时重置恢复追踪计数器（连胜次数归零）
- 记录降级时间戳（`tier_downgrade_at`）

### 升级（恢复）规则

恢复需要显式满足条件，不会因为回撤比例回落而自动升级。

| 恢复路径 | 条件 |
|------|------|
| halt → warning | 连续 3 笔盈利交易 |
| warning → caution | 连续 3 笔盈利交易 |
| caution → normal | 连续 3 笔盈利 **且** 当前回撤 < 5% |

### 双倍退防逻辑

升级后若连续遭遇 2 次亏损交易，强制退回上一级。

| 场景 | 操作 |
|------|------|
| 从 halt 恢复到 warning 后连亏 2 次 | 退回 halt |
| 从 warning 恢复到 caution 后连亏 2 次 | 退回 warning |
| 从 caution 恢复到 normal 后连亏 2 次 | 退回 caution |

> 设计目的：防止在市场环境未真正好转时过早恢复仓位，并切断"把亏的钱补回来"的报复性心理。

### 状态持久化

`risk_state` 表（key-value 结构）持久化以下状态：

| key | 含义 |
|------|------|
| `portfolio_hwm` | 账户权益历史最高点 |
| `risk_level` | 当前风控等级（normal/caution/warning/halt） |
| `consecutive_wins` | 当前连胜次数（用于恢复判定） |
| `losses_after_upgrade` | 升级后连亏次数（用于双倍退防） |
| `tier_downgrade_at` | 最近一次降级时间 |

### 更新时机

- 每次交易引擎执行后，从 Alpaca 获取账户权益并调用 `updateDrawdownTier()`
- 每笔交易完成后调用 `recordTradeOutcome()` 记录盈亏结果

### 实现文件

- `src/executors/risk-manager.ts` — `updateDrawdownTier()`, `checkRecovery()`, `recordTradeOutcome()`, `getRiskLevel()`, `getRiskMultiplier()`
- `src/db/schema.ts` — `riskState` 表

## 四、相关性惩罚

### 规则

当两个权益标的的 30 日日线相关性超过阈值时，两者的仓位乘数同时打折。

| 参数 | 值 |
|------|------|
| 相关性阈值 | 0.85 |
| 惩罚乘数 | ×0.7 |
| 检测的标的对 | SPY-QQQ, SPY-IWM, QQQ-IWM |

### 计算来源

相关性数据来自 `correlation_matrix` 分析结果（`analysis_results` 表），其中 `window_30d_daily` 字段存储了所有标的对的 30 日日线相关系数（如 `"SPY_IWM": 0.93`）。

### 惩罚效果

若 SPY-IWM 相关系数 = 0.93（> 0.85）：
- SPY 的 `sizeMultiplier` × 0.7（例如从 50% 降至 35%）
- IWM 的 `sizeMultiplier` × 0.7（同上）
- QQQ 不受影响（若其与 SPY/IWM 的相关性 < 0.85）

> 惩罚可叠加：若某标的同时与两个其他标的高度相关，乘数会连续打折。

### 设计目的

防止在系统性风险事件中，高度相关的标的同时亏损导致组合风险堆积。

### 不受惩罚的标的

- BTCUSD：独立的加密资产，相关性单独通过 BTC-SPY 相关性机制处理
- UUP：美元代理，走势逻辑独立

### 实现文件

- `src/executors/signal-scorer.ts` — `applyCorrelationPenalty()`, Layer 1 调整

## 五、1/4 凯利公式上限

### 公式

标准凯利公式：
```
f* = (b × p − q) / b
```

其中：
- p = 胜率（`biasCorrect = 1` 的比例）
- q = 1 − p
- b = 赔率比（平均盈利 ÷ 平均亏损）

实际使用 1/4 Kelly 作为保守上限：
```
实际仓位上限 = f* × 0.25
```

### 数据来源

从 `prediction_results` 表读取历史预测准确性数据：
- `biasCorrect`：market_bias 方向是否正确（1/0）
- `spyReturn`：对应 T+5 的 SPY 收益率

取最近 50 条记录计算。

### 激活条件

| 条件 | 要求 |
|------|------|
| 最小样本量 | ≥ 20 条有效记录 |
| 有效记录定义 | `biasCorrect IS NOT NULL AND spyReturn IS NOT NULL` |
| f* 为负 | 返回 0（完全不交易） |
| f* 为正 | 取 f*/4 作为 `sizeMultiplier` 上限 |

### 约束方式

作为 `sizeMultiplier` 的上限（cap），不影响方向判断和评分。如果某标的的仓位乘数已经低于 1/4 Kelly，则不受影响。

### 当前状态

系统运行初期（预测记录不足 20 条），Kelly 上限处于未激活状态。随着每日 T+5 评估积累数据，将在约 4 周后自动激活。

### 实现文件

- `src/executors/signal-scorer.ts` — `computeQuarterKelly()`, Layer 4 调整

## 六、L4 BTC 急跌联动

### 规则

当 BTC 24 小时跌幅超过 5%，自动减仓所有权益标的 20%。

| 参数 | 值 |
|------|------|
| 触发阈值 | BTC 24h 跌幅 ≥ -5% |
| 减仓比例 | 权益仓位 × 80%（保留 80%，卖出 20%） |
| 冷却期 | 12 小时 |
| 影响标的 | SPY, QQQ, IWM |
| 不影响 | BTCUSD（自身由 L1 止损保护）、UUP |

### BTC 价格计算

从 `hourly_prices` 表读取 BTCUSD 最近 25 根小时线，找到距当前 24 小时前最近的蜡烛线，计算涨跌幅。

### 执行流程

1. 检查 BTC 24h 收益率是否 < -5%
2. 检查是否在 12h 冷却期内（查询 `risk_events` 表 `event_type = "btc_crash_l4"`）
3. 遍历 Alpaca 账户中的 SPY/QQQ/IWM 持仓
4. 对每个持仓计算减仓数量（`currentQty × 20%`，向下取整）
5. 市价卖出减仓数量
6. 记录 `risk_events`（含 12h 冷却时间戳）
7. Slack 推送告警

### 检查频率

- 每小时整点（:00）与 L1 止损检查并行执行
- CLI 手动触发：`macro-sniper risk btc-crash`

### 实现文件

- `src/executors/risk-manager.ts` — `checkBtcCrashLinkage()`, `getBtc24hReturn()`

## 七、仓位调整层级总览

信号评分器（`scoreAllInstruments()`）对每个标的的仓位进行 4 层叠加调整，顺序执行：

```
基础评分 → ① 相关性惩罚 → ② ATR 仓位计算 → ③ 回撤乘数 → ④ Kelly 上限
```

| 层级 | 作用 | 触发条件 |
|------|------|------|
| Layer 1 | 相关性惩罚 | 权益标的对 30d corr > 0.85 |
| Layer 2 | ATR 仓位计算 | ATR 数据可用（≥15 日小时线） |
| Layer 3 | 回撤乘数 | 风控等级非 normal |
| Layer 4 | Kelly 上限 | ≥20 条预测记录 |

### 最终仓位计算

```
positionCap   = 账户权益 × 20%                    ← 比例上限
notionalTarget = min(positionCap, 风险预算 ÷ 止损%)  ← ATR 调整（数据不足时 = positionCap）
notionalFinal  = notionalTarget × sizeMultiplier   ← 最终下单金额
```

其中 `sizeMultiplier` 经过评分阈值、相关性惩罚、回撤乘数、Kelly 上限四重约束。

### 评分 → 仓位映射

**做多标的（SPY/QQQ/IWM/BTCUSD）：**

| 综合评分 | 方向 | sizeMultiplier |
|------|------|------|
| ≥ 50 | long | 1.0（100%） |
| ≥ 20 | long | 0.5（50%） |
| < 20 | flat | 0（不开仓） |

**双向标的（UUP）：**

| 综合评分 | 方向 | sizeMultiplier |
|------|------|------|
| ≥ 50 | long | 1.0 |
| ≥ 20 | long | 0.5 |
| −20 ~ 20 | flat | 0 |
| ≤ −20 | short | 0.5 |
| ≤ −50 | short | 1.0 |

### 否决机制

| 否决类型 | 条件 | 影响 |
|------|------|------|
| 信用否决（L3） | `credit_risk = risk_off_confirmed` | SPY/QQQ/IWM 强制 flat |
| BTC 同步否决 | `correlation = synchronized` 且 `market_bias = risk_off` | BTCUSD 强制 flat |
| market_bias 冲突 | `market_bias = conflicted` | 所有标的 sizeMultiplier 上限 0.75 |

### USD 驱动源差异化

USD 模型对权益标的的影响不再使用"美元强 = 股票压力大"的一刀切逻辑，而是根据**驱动 USD 强弱的具体因子**差异化处理。

#### 核心公式

原始逻辑：
```
equityImpact = (50 − composite) / 50 × sensitivity
```

新逻辑（加权混合）：
```
adjustedImpact = Σ(−deviation_i × modelWeight_i × impactMultiplier_i) / 50
equityImpact   = adjustedImpact × sensitivity
```

其中 `deviation_i = factorScore_i − 50`（正值 = 推动美元走强）。

当所有 `impactMultiplier` 均为 1.0 时，新公式等价于原始公式。

#### 驱动因子影响系数

| 驱动因子 | 条件 | impactMultiplier | 理由 |
|---------|------|-----------------|------|
| **利差支撑 (rate_support)** | yield_driver = real_rate | 0.5 | 经济强劲推高利率，企业盈利也好，USD 强但股票未必差 |
| | yield_driver = term_premium | 1.0 | 期限溢价驱动 = 财政/政策风险，全额传导 |
| | yield_driver = inflation | 0.8 | 通胀粘性，中等传导 |
| | yield_driver = unknown | 0.7 | 默认 |
| **风险溢价 (risk_premium)** | deviation > 0（全球避险） | 1.3 | 避险资金涌入美元 → 放大对股票的负面影响 |
| | deviation < 0（美国内部风险） | **−0.8** | **翻转**：USD 弱但股票也受伤（如 2025 关税冲击） |
| **便利性收益 (cy)** | — | 0.3 | 结构性安全溢价，与股票基本面关系不大 |
| **对冲传导 (hedge)** | — | 0.0 | 传导效率指标，不是方向信号 |
| **全球相对强弱 (global)** | — | 0.4 | 非美更差（被动走强）≠ 美国也差 |

#### DXY 敏感度系数（不变）

| 标的 | 敏感度 | 依据 |
|------|--------|------|
| QQQ | 1.0 | 海外收入占比 ~58% |
| SPY | 0.6 | 海外收入占比 ~40% |
| BTCUSD | 0.5 | 美元计价资产，中等敏感 |
| IWM | 0.2 | 小盘股以本土业务为主 |
| UUP | 0 | UUP 直接用 composite score，不经此逻辑 |

#### 示例对比

**场景：经济强劲，利差扩大（rate_support=75, risk_premium=55, 其他≈50, yield_driver=real_rate）**

| 方法 | composite | equityImpact (QQQ, sensitivity=1.0) |
|------|-----------|-------------------------------------|
| 原逻辑 | 58.75 | (50−58.75)/50 = −0.175 |
| 新逻辑 | 58.75 | −0.108（rate_support 的负面影响被 ×0.5 削弱） |

**场景：美国内部风险（rate_support=65, risk_premium=25, cy=40, yield_driver=term_premium）**

| 方法 | composite | equityImpact (QQQ) |
|------|-----------|---------------------|
| 原逻辑 | 46.75 | (50−46.75)/50 = +0.065（美元弱 = 利好股票 ❌） |
| 新逻辑 | 46.75 | −0.181（risk_premium 翻转：美元弱但股票也跌 ✓） |

#### 回退机制

当 USD 模型 metadata 不可用（数据库中无 `usd_model` 分析结果或字段缺失）时，自动回退到原始一刀切逻辑。

#### 实现文件

- `src/executors/signal-scorer.ts` — `buildUsdModelScore()`, `getDriverImpactMultiplier()`, `USD_DRIVER_IMPACT`
